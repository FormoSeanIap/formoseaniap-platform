This is the first entry in an 8-part series about building a personal platform from scratch — solo, on AWS, by pairing with an AI coding assistant for nearly every line of code, configuration, and infrastructure change.

## About This Series

This series, "Solo Platform Engineering: A Vibe Coding Diary," documents the full arc of building and shipping a portfolio platform as a solo engineer pairing with VS Code (Codex). Over eight parts, the diary covers infrastructure decisions and cost control on AWS, content migration from Medium, DNS traps during the first deploy, caching surprises and deploy gotchas, an architecture pivot to split the site into a professional engineering portfolio, a production cache collision incident, a multi-stage CSS debugging iteration, and finally, the limits of coding alone — the moments where human feedback mattered more than any prompt.

The goal is not a polished tutorial. It is a step-by-step account of what actually happened: the decisions that worked, the ones that did not, and the debugging process in between. If you have ever shipped something solo and wondered whether AI pairing could carry more of the weight, this series is for you.

## Why Build a Platform from Scratch?

The motivation was straightforward: I wanted a personal platform that reflected both engineering rigor and personal taste, and I did not want to compromise on either. Off-the-shelf solutions — static site generators with opinionated themes, hosted blogging platforms, drag-and-drop builders — all required giving up control over something that mattered to me, whether that was the hosting stack, the content pipeline, the design, or the deployment workflow.

Building from scratch meant owning every layer: how articles are stored and built, how the site is served and cached, how deploys happen, and how the design evolves. It also meant accepting the cost of that ownership — every bug, every misconfiguration, every late-night debugging session was mine to handle. The tradeoff was worth it, because the platform itself became the project, not just the content on it.

## Why AWS

AWS was the natural choice because the mental model was already there. Years of professional work on AWS meant I could think in terms of S3 buckets, CloudFront distributions, IAM roles, and Terraform resources without needing to translate concepts from another provider. Full control over the stack was non-negotiable — I wanted to understand every piece of the serving path from the origin bucket to the browser, and AWS gave me that transparency.

The cost question mattered early, not after launch. A personal platform with modest traffic should not cost more than a few dollars a month, and ideally less. I added a billing alarm from the start so that any accidental spending would surface before it grew. The combination of S3 for storage, CloudFront for CDN, and Terraform for infrastructure-as-code gave me a stack I could reason about, version-control, and keep cheap.

## The CloudFront Free Plan Discovery

The real cost-control enabler turned out to be a relatively new AWS feature: the CloudFront Free pricing plan. The Free plan includes 1 million HTTP/HTTPS requests per month, 100 GB of data transfer out, and — critically — no overage charges. If traffic exceeds the included limits, CloudFront simply stops serving until the next billing cycle rather than generating a surprise bill. For a personal platform, this was exactly the safety net I needed.

There was one catch. The CloudFront pricing mode that enables the Free plan is not yet supported by the AWS Go SDK, which means Terraform cannot manage it. The distribution itself is fully Terraform-managed, but switching the pricing mode to "Free" is a deliberate manual step in the AWS console. It is the one piece of the stack that is not infrastructure-as-code, and it will stay that way until the SDK catches up. I decided that one manual console click was an acceptable tradeoff for near-zero hosting cost.

## Pairing with VS Code (Codex)

The entire platform — infrastructure, build scripts, content pipeline, frontend, CI/CD — was built by pairing with VS Code (Codex). "Vibe coding" is the shorthand I use for this workflow: rather than writing every line from scratch, I described what I wanted, reviewed what the AI proposed, corrected course, and iterated. The AI handled the boilerplate and the first drafts; I handled the architecture decisions, the debugging, and the judgment calls.

In practice, this meant long conversations where I would describe a problem, paste in relevant files or error output, and ask for a targeted fix or a new module. The AI was remarkably good at generating Terraform resources, writing Python scripts, and producing HTML/CSS that was close to what I wanted. It was less good at holding the full picture of the repository in its head, and it was occasionally confident about things that were wrong.

## When AI Pairing Breaks Down

The failure modes were predictable in hindsight but frustrating in the moment. The most common was hitting token or context limits mid-conversation: I would be deep into a multi-file debugging session, and the AI would lose track of a file I had shared earlier, or forget a constraint I had stated three messages ago. Re-explaining the same context over and over was the single biggest source of friction.

The second failure mode was plausible but wrong suggestions. The AI would propose a fix that looked correct, used real API parameters, and matched the general shape of the solution — but was wrong in a specific detail that only surfaced at apply time or runtime. Hallucinated Terraform arguments, incorrect CloudFront Function syntax, and subtly wrong IAM policy conditions all made appearances. The danger was not that the suggestions were obviously bad; it was that they were convincing enough to skip verification, which I learned not to do.

## Tactics That Worked

Over time, I developed a set of habits that made the pairing significantly more productive. Smaller prompts worked better than large ones: instead of describing an entire feature, I would ask for one file at a time, or one function at a time. Asking for diffs instead of full file rewrites reduced the chance of the AI silently dropping lines I needed to keep. Pasting the exact error output — not a paraphrase, but the literal terminal output — consistently produced better fixes than describing the error in my own words.

Pinning requirements explicitly at the top of each conversation helped with context drift. I kept a few prompt templates that I reused across sessions: "Given this error output and these files, propose the minimal patch," "Review this Terraform plan output and flag anything that will fail at apply time," and "Here is the current state of this file — add only the function I describe, do not change anything else." These templates were not clever, but they reduced ambiguity and kept the AI focused on the right scope.

## Validation Workflow

Having automated checks was essential when my coding partner could not run the tests itself. Every change went through a small validation pipeline before I considered it done. `python3 scripts/terraform_validate_strict.py` caught Terraform syntax errors and deprecation warnings. `python3 scripts/build_articles.py` verified that the content pipeline could process every article without errors. Smoke checks against the live site confirmed that deploys actually landed. `python3 scripts/verify_production.py` ran a broader set of production health checks.

The discipline of running these checks after every AI-suggested change was what kept the project from accumulating silent regressions. The AI could not run the tests, but it could read the test output and propose fixes — so the workflow became a loop: propose a change, run the checks, paste the output back, iterate. It was not as fast as having a colleague who could run the tests themselves, but it was fast enough to ship.
